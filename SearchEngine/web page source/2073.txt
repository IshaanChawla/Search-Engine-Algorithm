(Continued)
&#x2022; Lazy
&#x2013; Many Stream operations are postponed until it is known how
much data is eventually needed
&#x2022; E.g., if you do a 10-second-per-item operation on a 100 element
list, then select the first entry, it takes 10 seconds, not 1000
seconds.
&#x2022; Parallelizable
&#x2013; If you designate a Stream as parallel, then operations on it will
automatically be done concurrently, without having to write
explicit multi-threading code
&#x2022; Can be unbounded
&#x2013; Unlike with collections, you can designate a generator
function, and clients can consume entries as long as they want,
with values being generated on the fly

    findFirst
&#x2022; Big idea
&#x2013; Returns an Optional for the first entry in the Stream. Since
Streams are often results of filtering, there might not be a first
entry, so the Optional could be empty.
&#x2022; There is also a similar findAny method, which might be faster for

or filter know to only find a single match and then stop.
&#x2022; Quick examples
&#x2013; When you know there is at least one entry
&#x2022; someStream.map(&#x2026;).findFirst().get()
&#x2013; When unsure if there are entries or not
&#x2022; someStream.filter(&#x2026;).findFirst().orElse(otherValue)

    Overview
&#x2022; Big idea
&#x2013; Streams defer doing most operations until you actually
need the results
&#x2022; Result
&#x2013; Operations that appear to traverse Stream multiple times
actually traverse it only once
&#x2013; Due to &#x201C;short-circuit&#x201D; methods, operations that appear to
traverse entire stream can stop much earlier.
&#x2022; stream.map(someOp).filter(someTest).findFirst().get()
&#x2013; Does the map and filter operations one element at a time.
Continues only until first match on the filter test.
&#x2022; stream.map(&#x2026;).filter(&#x2026;).filter(&#x2026;).allMatch(someTest)
&#x2013; Does the one map, two filter, and one allMatch test one element at
a time. The first time it gets false for the allMatch test, it stops.

        46.
      </a>
    Method Types: Overview
&#x2022; Intermediate methods
&#x2013; These are methods that produce other Streams. These methods
don&#x2019;t get processed until there is some terminal method called.
&#x2022; Terminal methods
&#x2013; After one of these methods is invoked, the Stream is
considered consumed and no more operations can be
performed on it.
&#x2022; These methods can do a side-effect (forEach) or produce a
value (findFirst)
&#x2022; Short-circuit methods
&#x2013; These methods cause the earlier intermediate methods to be
processed only until the short-circuit method can be evaluated.
&#x2022; Short-circuit methods can be intermediate (limit, skip) or terminal
(findFirst, allMatch)
&#x2013; E.g., this example only filters until it finds a single match:
Stream.of(someArray).filter(e -&gt; someTest(e)).findFirst().get()

If you thought of Streams as collections, you would think:
&#x2022; It would first call findGoogler on all 5 ids, resulting in 5 Employees
&#x2022; It would then call checkForNull on all 5 Employees
&#x2022; It would then call checkSalary on all remaining Employees
&#x2022; It would then get the first one (or null, if no Employees)
Instead, it builds a pipeline that, for each element in turn, calls
findGoogler, then checks that same element for null, then if non-null,
checks the salary of that same element, and if it exists, returns it.
