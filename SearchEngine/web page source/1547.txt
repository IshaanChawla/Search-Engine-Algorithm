<p>While <em>short int</em>, <em>long int</em>, and <em>long long int</em> are valid, the shorthand versions <em>short</em>, <em>long</em>, and <em>long long</em> should be preferred.  In addition to being less typing, adding the prefix <em>int</em> makes the type harder to distinguish from variables of type int.  This can lead to mistakes if the short or long modifier is inadvertently missed.</p>
<p><strong>Identifying integer</strong></p>
<p>Because the size of char, short, int, and long can vary depending on the compiler and/or computer architecture, it can be instructive to refer to integers by their size rather than name.  We often refer to integers by the number of bits a variable of that type is allocated (e.g. &#8220;32-bit integer&#8221; instead of &#8220;long&#8221;).</p>
<p><strong>Integer ranges and sign</strong></p>
<p>As you learned in the last section, a variable with n bits can store 2<sup>n</sup> different values.  But which specific values?  We call the set of specific values that a data type can hold its <strong>range</strong>.  The range of an integer variable is determined by two factors: its size (in bits), and its <strong>sign</strong>, which can be &#8220;signed&#8221; or &#8220;unsigned&#8221;.</p>

<p><strong>Overflow examples</strong></p>
<p>Consider a hypothetical unsigned variable that can only hold 4 bits.  Any of the binary numbers enumerated in the table above would fit comfortably inside this variable (because none of them are larger than 4 bits).</p>
<p>But what happens if we try to assign a value that takes more than 4 bits to our variable?  We get overflow: our variable will only store the 4 least significant (rightmost) bits, and the excess bits are lost.</p>
<p>For example, if we tried to put the decimal value 21 in our 4-bit variable:</p>
<table border=1 cellspacing=0 cellpadding=5>

<td style="text-align:right">10101</td>
</tr>

<p>21 takes 5 bits (<code>10101</code>) to represent.  The 4 rightmost bits (<code>0101</code>) go into the variable, and the leftmost (<code>1</code>) is simply lost.  Our variable now holds <code>0101</code>, which is the decimal value 5.</p>

<p>What do you think the result of this program will be?</p>
<pre>
x was: 65535
x is now: 0
</pre>
<p>What happened?  We overflowed the variable by trying to put a number that was too big into it (65536), and the result is that our value &#8220;wrapped around&#8221; back to the beginning of the range.</p>
<table border=1 cellspacing=0 cellpadding=3 width=100%>
<tr>
<td>
For advanced readers, here&#8217;s what&#8217;s actually happening behind the scenes: the number 65,535 is represented by the bit pattern <code>1111 1111 1111 1111</code> in binary.  65,535 is the largest number an unsigned 2 byte (16-bit) integer can hold, as it uses all 16 bits.  When we add 1 to the value, the new value should be 65,536.  However, the bit pattern of 65,536 is represented in binary as <code>1 0000 0000 0000 0000</code>, which is 17 bits!  Consequently, the highest bit (which is the 1) is lost, and the low 16 bits are all that is left.  The bit pattern <code>0000 0000 0000 0000</code> corresponds to the number 0, which is our result.
</td>
</tr>
</table>

<p>Overflow results in information being lost, which is almost never desirable.  If there is <em>any</em> doubt that a variable might need to store a value that falls outside its range, use a larger variable!</p>

<p><em>Rule: Do not depend on the results of overflow in your program</em>.</p>

		<p>That is actually a really difficult question. <img src="http://www.learncpp.com/blog/wp-includes/images/smilies/simple-smile.png" alt=":)" class="wp-smiley" style="height: 1em; max-height: 1em;" />  I have seen various programmers argue it either way, and there&#8217;s no clear answer.</p>
<p>Bjarne Stroustrup (who designed C++) says, &#8220;The unsigned integer types are ideal for uses that treat storage as a bit array. Using an unsigned instead of an int to gain one more bit to represent positive integers is almost never a good idea. Attempts to ensure that some values are positive by declaring variables unsigned will typically be defeated by the implicit conversion rules.&#8221;</p>
<p>I think Bjarne is correct on this one.</p>
<p>Using signed instead of unsigned integers even when you don&#8217;t expect negative numbers gives you a few benefits:</p>
<p>1) Many programmers use signed integers even when only dealing with positive numbers, because negative numbers can then be used as &#8220;error conditions&#8221;.  For example, it&#8217;s pretty common to write a function that is expected to return a positive number.  However, you can have it return a negative number if something goes wrong.  That way, the caller has a way of detecting something went astray.  (Note: You can also use exception handling as an alternative mechanism for returning errors)</p>
<p>2) What happens in this case:<br />
int foo(unsigned int nValue)<br />
{<br />
// something<br />
}</p>
<p>caller:<br />
int nValue = foo(-1);</p>
<p>The -1 gets silently converted into an unsigned integer (which would be a large positive number), and the function has no way of detecting that an invalid input was given to it.</p>
<p>3) If you expect a number to be positive, and your signed variable suddenly has a negative value, that&#8217;s a good indication your algorithm is wrong.</p>
<p>In short, as a rule of thumb, unless you have a good reason not to, it is better to use signed integers.</p>

		<p>Concrete basics of binary counting:<br />
If you understand that in decimal system 248 stands for<br />
2 x 10^2 +<br />
4 x 10^1 +<br />
8 x 10^0</p>
<p>and 13 for<br />
1 x 10^1 +<br />
3 x 10^0</p>
<p>then it is quite easy to convert in your head small decimals to binary and vice versa. For example 1011 is<br />
1 x 2^3 = 8 +<br />
0 x 2^2 = 0 +<br />
1 x 2^1 = 2 +<br />
1 x 2^1 = 1<br />
= 11.</p>
<p>When you go one integer (1) up in whatever the base (binary, octal, decimal etc.) you increase the lowest nominator to the highest possible until it reaches maximum, then you increase the second to the lowest if possible, if not, the third to the lowest etc. For example in octal system next from 4677 is 4700, because you can&#8217;t get higher than 7 (and after that of course 4701). In binary next from 1011 is 1100 (because from right to left first two 1&#8217;s can&#8217;t get higher). After 1100 -&gt; 1101, 1110, 1111. And you can assure it by counting:</p>
<p>1100 = 8 + 4 + 0 + 0 = 12<br />
1101 = 8 + 4 + 0 + 1 = 13<br />
1110 = 8 + 4 + 2 + 0 = 14<br />
1111 = 8 + 4 + 2 + 1 = 15.</p>

<p>So in this case, -1 is converted to an unsigned int by adding (UINT_MAX + 1) to the value.  The resultant value (UINT_MAX) is between 0 and UINT_MAX.  UINT_MAX is the maximum int (e.g. 65535).  So -1 maps to 65535.</p>

		<p>An an aside, it turns out the using two&#8217;s complement as the underlying representation makes this trivial.</p>
<p>Consider: -1 in two&#8217;s complement:<br />
binary representation for 1: 0000 0001<br />
flip the bits: 1111 1110<br />
add 1: 1111 1111 in two&#8217;s complement<br />
1111 1111 as an unsigned = 65535</p>
<p>Consider: -2 in two&#8217;s complement:<br />
binary representation for 2: 0000 0010<br />
flip the bits: 1111 1101<br />
add 1: 1111 1110 in two&#8217;s complement<br />
1111 1110 as an unsigned = 65534</p>
<p>Consider: -65535 in two&#8217;s complement (yes, this is outside the range for a 16-bit signed number, and should take 17 bits to represent properly in two&#8217;s complement, but we only have 16 bits, so lets use them and see what happens)<br />
binary representation for 65535: 1111 1111<br />
flip the bits: 0000 0000<br />
add 1: 0000 0001 in two&#8217;s complement<br />
0000 0001 as an unsigned = 1 (it still works!)</p>
<p>So if the compiler is using two&#8217;s complement binary representation for signed numbers (which many do), then all that&#8217;s needed is to interpret the number as an unsigned number.</p>

		<p>It is an integral data type, in the fact that it can only represent an integer, the same limitation that the other integral types (short, int, long) have.</p>
<p>The special handling of char traces back to the early days, when there were only 8 bits available to assign for text characters on a display screen.</p>
<p>You can do arithmetic with char types, you just have to be careful when you want to look at the results. </p>
<p><code>char a=65, b=66;<br />
    cout &lt;&lt; a &lt;&lt; b &lt;&lt; endl;<br />
    cout &lt;&lt; a+b &lt;&lt; endl;</code></p>
<p>Notice how cout implicitly cast the a+b evaluation to int type so you see the actual sum of 65+66 displayed, but when simple char variables are inserted into cout then the ASCII characters are output back to you.</p>
<p>If you are using char to write or read text, as most uses of it are, everything is cool as can be, in fact strings are simply of type char*</p>
