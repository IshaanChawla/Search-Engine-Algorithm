    <div class="post-text" itemprop="text">

<p>Someone came up with this proposed behavior recently and I haven't been able to find examples of services working the same way. I have to say I don't really like it, but can't put my finger on a precise reason.</p>

<p>It would work like this. We have a user in dB with:</p>

<ul>
<li>Email: "example@example.com"</li>
<li>Password: "12345678"</li>
</ul>

<p>The user goes to <code>example.com/sign_up</code> and signs up with:</p>

<ul>
<li>Email: "example@example.com"</li>
<li>Password: "12345678" (we might ask for a password confirmation)</li>
</ul>

<p>What we're doing now in this case is letting the user know there's a conflict; there's an account with that same email.</p>

<p>But since the user entered valid credentials, would it make any sense to authenticate him/her?</p>

<p>Edit: I don't think my question covers the same situation discussed <a href="http://ux.stackexchange.com/questions/74777">here</a>. These are not two different users with just the same email, it's one single user registering with the same credentials (both email <strong>and</strong> password) he/she used at an earlier time. What we usually see in this case is an unsuccessful registration attempt. The proposed behavior here is to check the credentials against the authentication and, if they work, simply log the user in.</p>

<p>As I said earlier, I've never seen this done and I don't like it myself. Some of the answers are giving very valid insight as to why it seems to be a bad idea.</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>So basically you want someone who signs up for a new account and enters already existing credentials, to log in as the owner of these credentials? </p>

<p>I wouldn't recommend this:</p>

<ol>
<li>The chance that the person signing up is not the owner of the existing account may be small, it is still possible.</li>
<li>The difference between signing up and logging in should be clear.</li>
<li>A user might forget it already has an account. Informing that an account with that e-mail address already exists is better than just logging in, which would be unexpected behavior and really confusing.</li>
</ol>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>I disagree with the other answers, and say <strong>yes</strong>, it may make sense (with a couple of caveats).</p>

<p>There is an increasing prevalence of the combined login/sign up form pattern on some sites, where the whole sign up form is simply email address and password, and all more substantive profile questions become an optional step after registration. This pattern dramatically lowers the barrier to entry for a site for new users.</p>

<p>If you apply this pattern, I think it does make sense to simply log the user in if they enter their credentials and happen to click the wrong button (provided the password was correct). The only problem I can see arises when the password <em>doesn't</em> match, and so your error message should be tailored to the action the user ostensibly requested (that is "that account could not be registered because an account already exists for your email address" as opposed to "invalid password" if the user clicked "Register New Account" but specified a different password).</p>

<p>All that seems like a pretty beneficial process from where I'm sitting, helping the user achieve their goal (accessing the site's restricted features) easily, but it <em>is</em> unusual and so you'd probably benefit from user testing this feature.</p>

<p>Worst case scenario, you could take a hybrid approach: present the conventional error screen ("An account with your email address already exists") and extend it with a link saying something like "Would you like to sign in now?"</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<h2>Yes, log the user in</h2>

<p>There are several ways an existing user might end up on a sign-up page:</p>

<ul>
<li>User clicks sign up by mistake</li>
<li>User recently signed up for an account and the browser URL autocomplete takes user back to that URL (most recent) </li>
<li>User forgot they signed up previously and is attempting to sign up again (and, like many users, ill-advisedly uses the same password on all sites)</li>
<li><p>Site landing page contains a sign up form and the user starts filling the form out (or browser credentials autocomplete fills out the form) </p>

<ul>
<li><p>eg  the default landing page for github.com has a signup form, and existing users who enter their email and password can log in via the sign up form:</p>

<p><img src="http://i.stack.imgur.com/BD0Z5.png" alt="github homme page"></p></li>
</ul></li>
</ul>

<p>Assuming that, like most sites, you don't allow multiple accounts with one email, then the user has provided you with valid login credentials so unless there is a non-ux reason (eg security concerns) it's okay to log the user in. </p>

<p>From the design perspective this is called <strong>interpolating user intent</strong>...you are making an informed guess as to what the user actually intends to do with the UX (in this case, by assuming that the user's ultimate goal is to access the site and signin/signup is just a means to that end) . </p>

<p>An example of interpolating user intent:</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>I would not recommend that. Signing up screen should inform visitor that a certain email is already registered. When you inform the user about that, he/she takes a step back to remember when and what he/she did that. This helps him/her get into context about the last visit to this site.</p>

<p>Your website should comply to user's mental model. I do not think every user will welcome system taking this proactive step of automatically logging the person in. It also helps create a different, not intentional workflow, which I would discourage.</p>

<p>Furthermore, many sites restrict login attempts on their login page. If you allow logging in via sign up pages, you would invite brute force attacks, and you'd have to put the restrictions on the sign up pages too. That is an overkill.</p>

<p>You can think of asking the user in a friendly note, that this email address is registered, do you want to log in instead?(then take him/her to login page) Or this email is already registered, have you forgotten a password?</p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p><strong>No.</strong></p>

<p>Irrespective of the fact that you may want to use something other than the e-mail address to uniquely identify a user (such as a separate user ID), no 2 different users can have the same e-mail address. Period. </p>

<p>If a user registers with an e-mail address that is already known by you, you may want to direct him/her to the "forgot password" functionality, or have him fill in another e-mail address. </p>

<p>There is <strong>no good use case</strong> in allowing 2 users to register with the same e-mail address. </p>
