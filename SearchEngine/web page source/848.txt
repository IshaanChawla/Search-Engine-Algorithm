    <div class="post-text" itemprop="text">

<p>Recently, a new vulnerability in Diffie-Hellman, informally referred to as 'logjam' has been published, for which <a href="https://weakdh.org/sysadmin.html">this page</a> has been put together suggesting how to counter the vulnerability:</p>

<blockquote>
  <p>We have three recommendations for correctly deploying Diffie-Hellman
  for TLS:</p>
  
  <ol>
  <li><strong>Disable Export Cipher Suites.</strong> Even though modern browsers no longer
  support export suites, the FREAK and Logjam attacks allow a
  man-in-the-middle attacker to trick browsers into using export-grade
  cryptography, after which the TLS connection can be decrypted. Export
  ciphers are a remnant of 1990s-era policy that prevented strong
  cryptographic protocols from being exported from United States. No
  modern clients rely on export suites and there is little downside in
  disabling them. </li>
  <li><strong>Deploy (Ephemeral) Elliptic-Curve Diffie-Hellman
  (ECDHE).</strong> Elliptic-Curve Diffie-Hellman (ECDH) key exchange avoids all
  known feasible cryptanalytic attacks, and modern web browsers now
  prefer ECDHE over the original, finite field, Diffie-Hellman. The
  discrete log algorithms we used to attack standard Diffie-Hellman
  groups do not gain as strong of an advantage from precomputation, and
  individual servers do not need to generate unique elliptic curves.</li>
  <li><strong>Generate a Strong, Unique Diffie Hellman Group</strong>. A few fixed groups are
  used by millions of servers, which makes them an optimal target for
  precomputation, and potential eavesdropping. Administrators should
  generate unique, 2048-bit or stronger Diffie-Hellman groups using
  "safe" primes for each website or server.</li>
  </ol>
</blockquote>

<p>What are the best-practice steps I should take to secure my server as per the above recommendations? </p>

<td class="answercell">
    <div class="post-text" itemprop="text">
<p>From the <a href="https://weakdh.org/sysadmin.html">article you linked</a>, there are three recommended steps to protect yourself against this vulnerability.</p>

<blockquote>
  <ol>
  <li><strong>Disable Export Cipher Suites</strong></li>
  </ol>
</blockquote>

<p>Dealt with in the configuration changes we'll make in 2. below (<code>!EXPORT</code> near the end of the <code>SSLCipherSuite</code> line is how we'll disable export cipher suites)</p>

<blockquote>
  <ol start="2">
  <li><strong>Deploy (Ephemeral) Elliptic-Curve Diffie-Hellman (ECDHE)</strong></li>
  </ol>
</blockquote>

<p>For this, you need to edit a few settings in your Apache config files - namely <code>SSLProtocol</code>, <code>SSLCipherSuite</code>, <code>SSLHonorCipherOrder</code> to have a "best-practices" setup. Something like the following will suffice:</p>

<pre><code>SSLProtocol             all -SSLv2 -SSLv3

SSLCipherSuite          ECDHE-RSA-AES128-GCM-SHA256:ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES256-GCM-SHA384:ECDHE-ECDSA-AES256-GCM-SHA384:DHE-RSA-AES128-GCM-SHA256:DHE-DSS-AES128-GCM-SHA256:kEDH+AESGCM:ECDHE-RSA-AES128-SHA256:ECDHE-ECDSA-AES128-SHA256:ECDHE-RSA-AES128-SHA:ECDHE-ECDSA-AES128-SHA:ECDHE-RSA-AES256-SHA384:ECDHE-ECDSA-AES256-SHA384:ECDHE-RSA-AES256-SHA:ECDHE-ECDSA-AES256-SHA:DHE-RSA-AES128-SHA256:DHE-RSA-AES128-SHA:DHE-DSS-AES128-SHA256:DHE-RSA-AES256-SHA256:DHE-DSS-AES256-SHA:DHE-RSA-AES256-SHA:AES128-GCM-SHA256:AES256-GCM-SHA384:AES128-SHA256:AES256-SHA256:AES128-SHA:AES256-SHA:AES:CAMELLIA:DES-CBC3-SHA:!aNULL:!eNULL:!EXPORT:!DES:!RC4:!MD5:!PSK:!aECDH:!EDH-DSS-DES-CBC3-SHA:!EDH-RSA-DES-CBC3-SHA:!KRB5-DES-CBC3-SHA

SSLHonorCipherOrder     on
</code></pre>

<p>Note: as for which <code>SSLCipherSuite</code> setting to use, this is always changing, and it is a good idea to consult resources such as <a href="https://wiki.mozilla.org/Security/Server_Side_TLS#Recommended_configurations">this one</a> to check for the latest recommended configuration.</p>

<blockquote>
  <p><strong>3. Generate a Strong, Unique Diffie Hellman Group</strong></p>
</blockquote>

<p>To do so, you can run</p>

<p><code>openssl dhparam -out dhparams.pem 2048</code>.</p>

<p>Note that this will put significant load on the server whilst the params are generated - you can always get around this potential issue by generating the params on another machine and using <code>scp</code> or similar to transfer them onto the server in question for use. </p>

<p>To use these newly-generated <code>dhparams</code> in Apache, from the <a href="https://httpd.apache.org/docs/current/ssl/ssl_faq.html">Apache Documentation</a>:</p>

<blockquote>
  <p>To generate custom DH parameters, use the openssl dhparam command.
  Alternatively, you can <strong>append the following</strong> standard 1024-bit DH
  parameters from RFC 2409, section 6.2 <strong>to the respective
  SSLCertificateFile file</strong>: </p>
</blockquote>

<p>(emphasis mine)</p>

<p>which is then followed by a standard 1024-bit DH parameter. From this we can infer that the custom-generated DH parameters may simply be appended to the relevant <code>SSLCertificateFile</code> in question.</p>

<p>To do so, run something similar to the following: </p>

<p><code>cat /path/to/custom/dhparam &gt;&gt; /path/to/sslcertfile</code></p>

<p>Alternatively, as per the <a href="https://weakdh.org/sysadmin.html">Apache subsection</a> of the article you originally linked, you may also specify the custom dhparams file you have created if you prefer not to alter the certificate file itself, thusly: </p>

<p><code>SSLOpenSSLConfCmd DHParameters "/path/to/dhparams.pem"</code></p>

<p>in whichever Apache config(s) are relevant to your particular SSL/TLS implementation - generally in <code>conf.d/ssl.conf</code>or <code>conf.d/vhosts.conf</code> but this will differ depending on how you have configured Apache.</p>

<p>It is worth noting that, as per <a href="https://wiki.mozilla.org/Security/Server_Side_TLS#Apache">this link</a>, </p>

<blockquote>
  <p>Before Apache 2.4.7, the DH parameter is always set to 1024 bits and
  is not user configurable. This has been fixed in mod_ssl 2.4.7 that
  Red Hat has backported into their RHEL 6 Apache 2.2 distribution with
  httpd-2.2.15-32.el6</p>
</blockquote>

<p>You should check whether your Apache version is later than these version numbers depending on your distribution, and if not - update it if at all possible. </p>

<p>Once you have performed the above steps to update your configuration, and restarted the Apache service to apply the changes, you should check that the configuration is as-desired by running the tests on <a href="https://www.ssllabs.com/ssltest/">SSLLabs</a> and on <a href="https://weakdh.org/sysadmin.html">the article</a> related to this particular vulnerability. </p>
