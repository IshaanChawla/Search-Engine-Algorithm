The Java environment provides a Java Servlets interface and support for standard interfaces to the App Engine scalable datastore and services, such as JDO, JPA, JavaMail, and JCache. Standards support makes developing your application easy and familiar, and also makes porting your application to and from your own servlet environment straightforward.
<aside class="special">The Java runtime is also available in a <a href="/appengine/docs/managed-vms/">Managed VM</a> hosting environment.</aside></p>

<h2 id="Java_Introduction">Introduction</h2>



<p>App Engine runs your Java web application using a Java 7 JVM in a safe "sandboxed" environment. App Engine invokes your app's servlet classes to handle requests and prepare responses in this environment.</p>
<p>The <a href="/appengine/docs/java/tools/eclipse">Google Plugin for Eclipse</a> adds new project wizards and debug configurations to your <a href="http://www.eclipse.org/">Eclipse</a> IDE for App Engine projects. App Engine for Java makes it especially easy to develop and deploy world-class web applications using <a href="/web-toolkit/">Google Web Toolkit</a> (GWT). The Eclipse plugin comes bundled with the App Engine and GWT SDKs.</p>
<p>Third-party plugins are available for other Java IDEs as well. For example, for NetBeans see <a href="https://code.google.com/p/nb-gaelyk-plugin/downloads/list">NetBeans plugin for Gaelyk Framework</a>. For IntelliJ, see <a href="http://plugins.intellij.net/plugin/?id=4254">Google App Engine Integration for IntelliJ</a>. (These links take you to third-party resources.)
Warning: Applications that use Java 6 need to be migrated to Java 7.  Existing applications that use the Java 6 runtime will continue to work, but updating these applications will require a <a href="http://goo.gl/ycffXq">whitelist request</a>.</p>
<p>App Engine uses <a href="https://jcp.org/aboutJava/communityprocess/mrel/jsr154/index.html">the Java Servlet 2.5 standard</a> for web applications. You provide your app's servlet classes, JavaServer Pages (JSPs), static files and data files, along with the deployment descriptor (the <code>web.xml</code> file) and other configuration files, in a standard WAR directory structure. App Engine serves requests by invoking servlets according to the deployment descriptor.</p>
<p>The secured "sandbox" environment isolates your application for service and security. It ensures that apps can only perform actions that do not interfere with the performance and scalability of other apps. For instance, an app cannot spawn threads in some ways, write data to the local file system or make arbitrary network connections. An app also cannot use JNI or other native code. The JVM can execute any Java bytecode that operates within the sandbox restrictions.</p>
<p>The App Engine platform provides many <a href="/appengine/docs/java/apis">services</a> that your code can call. Your application can also configure <a href="/appengine/docs/java/config/cron">scheduled tasks</a> that run at specified intervals.</p>
<p>If you haven't already, see <a href="/appengine/docs/java/gettingstarted">the Java Getting Started Guide</a> for an interactive introduction to developing web applications with Java technologies and Google App Engine.</p>



<h2 id="Java_Selecting_the_Java_runtime">Selecting the Java runtime</h2>



<p>App Engine knows to use the Java runtime environment for your application when you use the AppCfg tool from the Java SDK to upload the app.</p>
<p>The App Engine Java API is represented by the <code>appengine-api-*.jar</code> included with the SDK (where <code>*</code> represents the version of the API and the SDK). You select the version of the API your application uses by including this JAR in the application's <code>WEB-INF/lib/</code> directory. If a new version of the Java runtime environment is released that introduces changes that are not compatible with existing apps, that environment will have a new version number. Your application will continue to use the previous version until you replace the JAR with the new version (from a newer SDK) and re-upload the app.</p>




<h2 id="Java_The_sandbox">The sandbox</h2></p>
<p>To allow App Engine to distribute requests for applications across multiple web servers, and to prevent one application from interfering with another, the application runs in a restricted "sandbox" environment. In this environment, the application can execute code, store and query data in the App Engine datastore, use the App Engine mail, URL fetch and users services, and examine the user's web request and prepare the response.</p>
<p>An App Engine application cannot:</p>
<ul>
<li>
<p>write to the filesystem. Applications must use
<a href="/appengine/docs/java/datastore">the App Engine datastore</a> for storing persistent data.  Reading from the filesystem is allowed, and all application files uploaded with the application are available.</p>
</li>
<li>
<p>respond slowly. A web request to an application must be handled  within a few seconds. Processes that take a very long time to respond are terminated to avoid overloading the web server.</p>
</li>
<li>
<p>make other kinds of system calls.</p>
</li>
</ul>


<h3 id="threads">Threads</h3>
<p>A Java application can create a new thread, but there are some restrictions on how to do it. These threads can't "outlive" the request that creates them. (On a backend server, an application can spawn a <a href="/appengine/docs/java/backends/#Java_Background_threads">background thread</a>, a thread that can "outlive" the request that creates it.)</p>
<p>An application can</p>
<ul>
<li>Implement <code>java.lang.Runnable</code>; and</li>
<li>Create a thread factory by calling <a href="/appengine/docs/java/javadoc/com/google/appengine/api/ThreadManager#currentRequestThreadFactory()"><code>com.google.appengine.api.ThreadManager.currentRequestThreadFactory()</code></a></li>
<li>call the factory's <code>newRequestThread</code> method, passing in the <code>Runnable</code>, <code>newRequestThread(runnable)</code></li>
</ul>
<p>or use the factory object returned by <a href="/appengine/docs/java/javadoc/com/google/appengine/api/ThreadManager#currentRequestThreadFactory()"><code>com.google.appengine.api.ThreadManager.currentRequestThreadFactory()</code></a> with an <code>ExecutorService</code> (e.g., call <code>Executors.newCachedThreadPool(factory)</code>).</p>
<p>However, you must use one of the methods on <code>ThreadManager</code> to create your threads. You cannot invoke <code>new Thread()</code> yourself or use the <a href="http://docs.oracle.com/javase/6/docs/api/java/util/concurrent/Executors.html#defaultThreadFactory">default thread factory</a>.</p>
<p>An application can perform operations against the current thread, such as <code>thread.interrupt()</code>.</p>
<p>Each request is limited to 50 concurrent request threads. The Java runtime will throw a
<code>java.lang.IllegalStateException</code> if you try to create more than 50 threads in a single request.</p>
<aside class="caution"><strong>Caution:</strong><span> Threads are a powerful feature that are full of surprises. If you aren't comfortable with using threads with Java, we recommend Goetz, <em>Java Concurrency in Practice</em>.</span></aside>
<h3 id="the_filesystem">The Filesystem</h3>
<p>A Java application cannot use any classes used to write to the filesystem, such as <code>java.io.FileWriter</code>. An application can read its own files from the filesystem using classes such as <code>java.io.FileReader</code>. An application can also access its own files as "resources", such as with <code>Class.getResource()</code> or <code>ServletContext.getResource()</code>.</p>
<p>Only files that are considered "resource files" are accessible to the application via the filesystem. By default, all files in the WAR are "resource files."  You can exclude files from this set using the <a href="/appengine/docs/java/config/appconfig">appengine-web.xml</a> file.</p>
<h3 id="javalangsystem">java.lang.System</h3>
<p>Features of the <code>java.lang.System</code> class that do not apply to App Engine are disabled.</p>
<p>The following <code>System</code> methods do nothing in App Engine: <code>exit()</code>, <code>gc()</code>, <code>runFinalization()</code>, <code>runFinalizersOnExit()</code></p>
<p>The following <code>System</code> methods return <code>null</code>: <code>inheritedChannel()</code>, <code>console()</code></p>
<p>An app cannot provide or directly invoke any native JNI code. The following <code>System</code> methods raise a <code>java.lang.SecurityException</code>: <code>load()</code>, <code>loadLibrary()</code>, <code>setSecurityManager()</code></p>
<h3 id="reflection">Reflection</h3>
<p>An application is allowed full, unrestricted, reflective access to its own classes.</p>
<p>It may query any private members, call the method <code>java.lang.reflect.AccessibleObject.setAccessible()</code>, and read/set private members.</p>
<p>An application can also reflect on JRE and API classes, such as <code>java.lang.String</code> and <code>javax.servlet.http.HttpServletRequest</code>. However, it can only access public members of these classes, not protected or private.</p>
<p>An application cannot reflect against any other classes not belonging to itself, and it can not use the <code>setAccessible()</code> method to circumvent these restrictions.</p>
<h3 id="custom_class_loading">Custom class loading</h3>
<p>Custom class loading is fully supported under App Engine. An application is allowed to define its own subclass of ClassLoader that implements application-specific class loading logic. Please be aware, though, that App Engine overrides all ClassLoaders to assign the same permissions to all classes loaded by your application. If you perform custom class loading, be cautious when loading untrusted third-party code.</p>



<h2 id="Java_Class_loader_JAR_ordering">Class loader JAR ordering</h2>

<p>Sometimes, it may be necessary to redefine the order in which JAR files are scanned for classes in order to resolve collisions between class names. In these cases, loading priority can be granted to specific JAR files by adding a <code>&lt;class-loader-config&gt;</code> element containing <code>&lt;priority-specifier&gt;</code> elements in the <code>appengine-web.xml</code> file. For example:</p>
<pre class="prettyprint notranslate" translate="no"><code>&lt;class-loader-config&gt;
	&lt;priority-specifier filename="mailapi.jar"/&gt;
&lt;/class-loader-config&gt;
</code></pre>
<p>This places "<code>mailapi.jar</code>" as the first JAR file to be searched for classes, barring those in the directory <code>war/WEB-INF/classes/</code>.</p>
<p>If multiple JAR files are prioritized, their original loading order (with respect to each other) will be used. In other words, the order of the <code>&lt;priority-specifier&gt;</code> elements themselves does not matter.</p>
<h2 id="Java_The_JRE_white_list">The JRE white list</h2>

<p>Access to the classes in the Java standard library (the Java Runtime Environment, or JRE) is limited to the classes in <a href="/appengine/docs/java/jrewhitelist">the App Engine JRE White List</a>.</p>
<h2 id="Java_No_signed_JAR_files">No signed JAR files</h2>

<p>App Engine's precompilation isn't compatible with signed JAR files. If your application is precompiled (the default), it can't load signed JAR files. If the application tries to load a signed JAR, at runtime App Engine will generate an exception like</p>
<pre class="prettyprint notranslate" translate="no"><code>java.lang.SecurityException: SHA1 digest error for com/example/SomeClass.class
    at com.google.appengine.runtime.Request.process-d36f818a24b8cf1d(Request.java)
    at sun.security.util.ManifestEntryVerifier.verify(ManifestEntryVerifier.java:210)
    at java.util.jar.JarVerifier.processEntry(JarVerifier.java:218)
    at java.util.jar.JarVerifier.update(JarVerifier.java:205)
    at java.util.jar.JarVerifier$VerifierStream.read(JarVerifier.java:428)
    at sun.misc.Resource.getBytes(Resource.java:124)
    at java.net.URLClassLoader.defineClass(URLClassLoader.java:273)
    at sun.reflect.GeneratedMethodAccessor5.invoke(Unknown Source)
    at sun.reflect.DelegatingMethodAccessorImpl.invoke(DelegatingMethodAccessorImpl.java:43)
    at java.lang.reflect.Method.invoke(Method.java:616)
    at java.lang.ClassLoader.loadClass(ClassLoader.java:266)
</code></pre>
<p>There are two ways to work around this:</p>
<ul>
<li><a href="https://www.google.com/search?q=java+unsign+jar&amp;nfpr=1">Strip the JAR's signature</a>; <strong>or</strong></li>
<li><a href="./config/appconfig#Java_appengine_web_xml_Disabling_precompilation">Disable precompilation for the whole application</a>.</li>
</ul>



<h2 id="Java_tools">The Java SDK and tools</h2>



<p>The <a href="/appengine/downloads#Google_App_Engine_SDK_for_Java">App Engine Java SDK</a>
includes tools for testing your application, uploading your application files,
and downloading log data. The SDK also includes a component for <a href="/appengine/docs/java/tools/ant">Apache
Ant</a> to simplify tasks common to App Engine
projects. The <a href="/appengine/docs/java/tools/eclipse">Google Plugin for Eclipse</a>
adds features to the Eclipse IDE for App Engine development, testing and
deployment, and includes the complete App Engine SDK. The Eclipse plugin also
makes it easy to develop <a href="/web-toolkit/">Google Web Toolkit</a> applications and
run them on App Engine.</p>
<p>The App Engine Java SDK also has a plugin for supporting development with <a href="/appengine/docs/java/tools/maven">Apache Maven</a>.</p>
<p>The <a href="/appengine/docs/java/tools/devserver">development server</a> runs your application on your local computer for development and testing. The server simulates the App Engine datastore, services and sandbox restrictions. The development server can also generate configuration for datastore indexes based on the queries the app performs during testing.</p>
<p>A multipurpose tool called <a href="/appengine/docs/java/tools/uploadinganapp">AppCfg</a> handles all command-line interaction with your application running on App Engine. AppCfg can upload your application to App Engine, or just update the datastore index configuration so you can build new indexes before updating the code. It can also download the app's log data, so you can analyze your app's performance using your own tools.</p>
