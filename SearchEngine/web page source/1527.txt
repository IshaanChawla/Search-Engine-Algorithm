<p>All of these comments give the reader a good idea of what the program is trying to accomplish without having to look at the actual code.  The user (possibly someone else, or you if you&#8217;re trying to reuse code you&#8217;ve already previously written) can tell at a glance whether the code is relevant to what he or she is trying to accomplish.  This is particularly important when working as part of a team, where not everybody will be familiar with all of the code.</p>

<p>Finally, comment should be written in a way that makes sense to someone who has no idea what the code does.  It is often the case that a programmer will say &#8220;It&#8217;s obvious what this does!  There&#8217;s no way I&#8217;ll forget about this&#8221;.  Guess what?  It&#8217;s not obvious, and you will be amazed how quickly you forget. <img src="http://www.learncpp.com/blog/wp-includes/images/smilies/simple-smile.png" alt=":)" class="wp-smiley" style="height: 1em; max-height: 1em;" />  You (or someone else) will thank you later for writing down the what, how, and why of your code in human language.  Reading individual lines of code is easy.  Understanding what goal they are meant to accomplish is not.</p>
<p>To summarize:</p>
<ul>
<li>At the library, program, or function level, describe <em>what</em>
<li>Inside the library, program, or function, describe <em>how</em>
<li>At the statement level, describe <em>why</em>.
</ul>

<p>There are quite a few reasons you might want to do this:</p>
<p>1) You&#8217;re working on a new piece of code that won&#8217;t compile yet, and you need to run the program.  The compiler won&#8217;t let you run if there are compiler errors.  Commenting out the code that won&#8217;t compile will allow the program to compile so you can run it.  When you&#8217;re ready, you can uncomment the code, and continue working on it.</p>
<p>2) You&#8217;ve written code that compiles but doesn&#8217;t work correctly, and you don&#8217;t have time to fix it until later.  Commenting out the broken code will ensure the broken code doesn&#8217;t execute and cause problems until you can fix it.</p>
<p>3) To find the source of an error.  If a program isn&#8217;t producing the desired results (or is crashing), it can be useful in some cases to disable parts of your code to see if you can isolate what&#8217;s causing it to not work correctly.  If you comment out one or more lines of code, and your program starts working as expected (or stops crashing), odds are whatever you last commented out was part of the problem.  You can then investigate why those lines of code are causing the problem.</p>
<p>4) You want to replace one piece of code with another piece of code.  Instead of just deleting the original code, you can comment it out and leave it there for reference until you&#8217;re sure your new code works properly.  Once you are sure your new code is working, you can remove the old commented out code.  If you can&#8217;t get your new code to work, you can always delete the new code and uncomment the old code to revert back to what you had before.</p>
<p>Commenting out code is a common thing to do while developing, so many IDEs provide support for commenting out a highlighted section of code.  How you access this functionality varies by IDE (in Visual Studio, you can find it in the Edit menu under Edit->Advanced->Comment Selection/Uncomment Selection).</p>
<p><strong>A note on comments in these tutorials</strong></p>
<p>Throughout the rest of this tutorial series, I&#8217;ll be using comments inside code blocks to help illustrate how things work.  Astute readers will note that by the above standards, most of these comments are horrible. <img src="http://www.learncpp.com/blog/wp-includes/images/smilies/simple-smile.png" alt=":)" class="wp-smiley" style="height: 1em; max-height: 1em;" />  As you read through the rest of the tutorials, keep in mind that the comments are serving an intentional educational purpose, not trying to demonstrate what good comments look like.</p>
