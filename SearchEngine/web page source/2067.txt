    Main Points
&#x2022; Lambdas are lexically scoped
&#x2013; They do not introduce a new level of scoping
&#x2022; Implications
&#x2013; The &#x201C;this&#x201D; variable refers to the outer class, not to the
anonymous inner class that the lambda is turned into
&#x2013; There is no &#x201C;OuterClass.this&#x201D; variable
&#x2022; Unless lambda is inside a normal inner class
&#x2013; Lambdas cannot introduce &#x201C;new&#x201D; variables with same
name as variables in method that creates the lambda
&#x2022; However, lambdas can refer to (but not modify) local
variables from the surrounding code
&#x2013; Lambdas can still refer to (and modify) instance variables
from the surrounding class

    Aside: Default Methods
&#x2022; Question: multiple methods in SAM interfaces
&#x2013; Function has &#x201C;apply&#x201D;, so how can it also have compose, andThen, and
identity?
&#x2013; Isn&#x2019;t the whole point of functional (SAM) interfaces that they have a single
method?
&#x2022; Answer: default and static methods
&#x2013; Functional interfaces have a single abstract method (one that must be
defined in class that implements interface). That is the method that the
lambda specifies. But, interfaces in Java 8 can have default methods that
have real method bodies and are inherited. Java 8 interfaces can also have
static methods.
&#x2022; This makes Java 8 interfaces more like abstract classes. Now we can have
multiple inheritance of implementation (mixins). This also means that, to
avoid the &#x201C;diamond problem&#x201D;, you can no longer necessarily implement as
many interfaces as you want. In particular, if a class implements two
interfaces that have default methods with the same name, it must override
that method or it won&#x2019;t compile. In your own code, you can also use
InterfaceName.super.methodName(&#x2026;) to disambiguate.

The is the one and only method specification. As with interface methods in previous
Java versions, it must be defined in classes that implement the interface.
These are default methods, a new feature of Java 8. Classes that implement
the interface do not need to implement these methods, but will inherit both
their names and their behavior. The classes can also override them. Interfaces
are prohibited from having default methods for methods of Object (e.g., toString, equals).
Interfaces in Java 8 can now have static methods.

    Methods from Function
&#x2022; compose
&#x2013; f1.compose(f2) means to first run f2, then pass the result to f1.
Default method.
&#x2022; Of course, you cannot really &#x201C;call&#x201D; lambdas. So, strictly speaking,
f1.compose(f2) means to produce a Function whose apply
method, when called, first passes the argument to the apply
method of f2, then passes the result to the apply method of f1.
&#x2022; andThen
&#x2013; f1.compose(f2) means to first run f1, then pass the result to f2.
So, f2.andThen(f1) is the same as f1.compose(f2). Math
people usually think of &#x201C;compose&#x201D; instead of &#x201C;andThen&#x201D;.
Default method.
&#x2022; identity
&#x2013; Function.identity() creates a function whose apply method just
returns the argument unchanged. Static method.

&#x2022; Idea

public static &lt;T&gt; List&lt;T&gt; transform2(List&lt;T&gt; origValues, Function&lt;T,T&gt; ... transformers) {
Function&lt;T,T&gt; composedFunction = composeAll(transformers);

&#x2022; Helper method
public static &lt;T&gt; Function&lt;T,T&gt; composeAll(Function&lt;T,T&gt; ... functions) {
Function&lt;T,T&gt; result = Function.identity();
for(Function&lt;T,T&gt; f: functions) { result = result.compose(f) ; }
return(result);
}

    Typing Issues with compose
and Method References
&#x2022; Legal: two steps
Function&lt;Double,Double&gt; round = Math::rint;
transform(nums, round.compose(Math::sqrt));
&#x2022; Illegal: one step
transform(nums, Math::rint.compose(Math::sqrt));
&#x2022; Question: why?
&#x2013; It looks like the same code both times
&#x2022; Answer: typing
&#x2013; Math::rint does not have a type until it is assigned to a
variable or passed to a method. Any interface that has a
single (abstract) method that can take two doubles could
be the target for Math::rint.
&#x2022; But, those other interfaces do not have &#x201C;compose&#x201D; method27

    Method from Consumer
&#x2022; andThen
&#x2013; f1.andThen(f2) produces a Consumer that first passes
argument to f1 (i.e., to its accept method), then passes
argument to f2. Default method.
&#x2022; Note the difference between &#x201C;andThen&#x201D; of
Consumer and of Function.
&#x2013; With andThen from Consumer, the argument is passed to
the accept method of f1, then that same argument is
passed to the accept method of f2.
&#x2013; With andThen from Function, the argument is passed to
the apply method of f1, and then the result of apply is
passed to the apply method of f2.

&#x2013; Return a Predicate, Function, or other lambda from a

&#x2013; Use a regular method
&#x2022; Use normal &#x201C;return&#x201D; syntax, but have a lambda as the
return value
&#x2013; Use a Function
&#x2022; Technically, this is a regular method (apply). But, you can
use &#x201C;double&#x201D; lambda syntax: a lambda whose expression

    Variable Scoping: No New Level
(&#x201C;this&#x201D; is Surrounding Class)
&#x2022; Illegal: repeated variable name
double x = 1.2;
someMethod(x -&gt; doSomethingWith(x));
&#x2022; Illegal: repeated variable name
double x = 1.2;
someMethod(y -&gt; { double x = 5.6; &#x2026; });
&#x2022; Illegal: lambda modifying local var from the outside
double x = 1.2;
someMethod(y -&gt; x = 3.4);
&#x2022; Legal: modifying instance variable
private double x = 1.2;
public void foo() { someMethod(y -&gt; x = 3.4); }
&#x2022; Legal: local name matching instance variable name
private double x = 1.2;
public void bar() { someMethod(x -&gt; x + this.x); }
