<p>Try compiling this program and running it for yourself.  When you run the program, it will print &#8220;Enter a number: &#8221; and then wait for you to enter one.  Once you enter a number (and press enter), it will print &#8220;You entered &#8221; followed by the number you just entered.</p>
<p>For example (I entered 4):</p>
<pre>
Enter a number: 4
You entered 4
</pre>
<p>This is an easy way to get input from the user, and we will use it in many of our examples going forward.</p>
<p>If your screen closes immediately after entering a number, please see section <a href="http://www.learncpp.com/cpp-tutorial/07-a-few-common-cpp-problems/">0.7 -- a few common cpp problems</a> for a solution.</p>
<p>(As an interesting side note, if you enter a really big number, the results may surprise you.  Try it!  This happens because x can only hold numbers up to a certain size.  After that, it &#8220;overflows&#8221;.  We&#8217;ll discuss overflow in a future section.)</p>
<p><strong>std::cin, std::cout, &lt;&lt;, and &gt;&gt;</strong></p>
<p>New programmers often mix up std::cin, std::cout, &lt;&lt; and &gt;&gt;.  Here&#8217;s an easy way to remember:</p>
<ul>
<li>std::cin and cout always go on the left-hand side of the statement.</li>
<li>std::cout is used to output a value (cout = output)</li>
<li>std::cin is used to get an input value (cin = input)</li>
<li>&lt;&lt; is used with std::cout, and shows the direction that data is moving from the r-value to the console.  <code>std::cout << 4 moves the value of 4 to the console</code></li>
<li>&gt;&gt; is used with std::cin, and shows the direction that data is moving from the console into the variable.  <code>std::cin >> x moves the value from the console into x</code></li>
</ul>
<p><strong>The std namespace</strong></p>
<p>Everything in the standard library lives inside a special container (called a <strong>namespace</strong>) that is named <em>std</em> (short for standard).</p>
<p>It turns out that std::cout's name isn't really "std::cout".  It's actually just "cout", and "std" is the name of the namespace it lives inside.</p>
<p>We'll talk more about namespaces in a future lesson and also teach you how to create your own.  For now, the only thing you need to know about namespaces is that whenever we use something (like cout) that is part of the standard library, we need to tell the compiler that it is inside the std namespace.</p>
<p><strong>Explicit namespace qualifier std::</strong></p>
<p>One way to tell the compiler that cout lives in the std namespace is by using the "std::" prefix. For example:</p>

<p>The using declaration "using std::cout;" tells the compiler that we're going to be using the object "cout" from the std namespace.  So whenever it sees "cout", it should assume that we mean "std::cout".  Therefore on line 6, we can type "cout" instead of "std::cout".</p>
<p>This doesn't save much effort in this trivial example, but if you are using cout (or cin, or endl) a lot inside of a function, using a using declaration can make your code more readable.</p>
<p>Although this method is less explicit than using the "std::" prefix, it's generally considered safe and acceptable.</p>

<p></p>
<p>The using directive "using namespace std;" tells the compiler that we want to use <em>everything</em> in the std namespace, so if the compiler finds a name it doesn't recognize, it will check the std namespace.  Consequently, when the compiler encounters "cout" (which it won't recognize), it'll look in the std namespace and find it there.</p>
<p>There's open debate about whether this solution is good practice or not.  Because using directives pull in all of the names in a namespace, the chance for a naming collisions goes up significantly (but is still small).</p>
<p>Many people avoid "using directives" altogether.  Others find them acceptable so long as they are used <em>only</em> within individual functions, which helps minimize the amount of "naming pollution".</p>
<p>In many of the examples beyond this page, we've opted to use "using directives" within functions.  It is up to you whether you want to follow this practice yourself, or avoid using them altogether.  In either case, you should avoid using a "using directives" outside of a function body.</p>
<p><em>Rule: Avoid using directives outside of a function body.</em></p>

		<p>It&#8217;s no problem.</p>
<p>If the objects in your own library are not in a namespace, just call them directly (e.g. doSomething()).</p>
<p>If the objects in your own library are in a namespace, either call them directly (e.g. myNamespace::doSomething()) or use a using statement for that library (eg. using namespace myNamespace) and then call them directly (e.g. doSomething()).</p>
<p>As long as the names of the objects in your library are different than those in the standard library, you won&#8217;t have any problems.</p>
<p>And even if you do encounter a naming collision, there are ways of working around it.  For example, you can remove the using statements and use explicit qualifiers (e.g. std:: or myNamespace::).</p>

<p>Decided to take the basic concept a step further with the std::cin statement test:</p>
<p>int main()<br />
{<br />
&nbsp;&nbsp;&nbsp;&nbsp;// set cout to std::cout because I&apos;m lazy :: also test that this statement works properly<br />
&nbsp;&nbsp;&nbsp;&nbsp;using std::cout; </p>
<p>&nbsp;&nbsp;&nbsp;&nbsp;// test to make sure cin works the way cout does<br />
&nbsp;&nbsp;&nbsp;&nbsp;using std::cin;</p>
<p>&nbsp;&nbsp;&nbsp;&nbsp;/* Also using only individual commands :: I don&apos;t trust automating more than absolutely neccessary.<br />
&nbsp;&nbsp;&nbsp;&nbsp;* Computers are stupid, don&apos;t entrust them with anything that may involve making a decision.<br />
&nbsp;&nbsp;&nbsp;&nbsp;* Since &quot;using namespace std;&quot; requires the computer to make a decision if two commands overlap,<br />
&nbsp;&nbsp;&nbsp;&nbsp;* I&apos;m simply not willing to entrust it with a decision because, with my luck, it&apos;ll make the wrong call. */</p>
<p>&nbsp;&nbsp;&nbsp;&nbsp;// Also, the above is a test of enclosed comments across multiple lines.</p>
<p>&nbsp;&nbsp;&nbsp;&nbsp;// set X and Y variables first, prefer to set early before messing with anything else</p>
<p>&nbsp;&nbsp;&nbsp;&nbsp;// defaults X so it doesn&apos;t break when called<br />
&nbsp;&nbsp;&nbsp;&nbsp;int x = 0;</p>
<p>&nbsp;&nbsp;&nbsp;&nbsp;// defaults Y so it doesn&apos;t break when called<br />
&nbsp;&nbsp;&nbsp;&nbsp;int y = 0;</p>
<p>&nbsp;&nbsp;&nbsp;&nbsp;// ask user for a number,<br />
&nbsp;&nbsp;&nbsp;&nbsp;cout &lt;&lt; &quot;Enter a number:&quot;;<br />
&nbsp;&nbsp;&nbsp;&nbsp;<br />
&nbsp;&nbsp;&nbsp;&nbsp;// read number from console and store it in x<br />
&nbsp;&nbsp;&nbsp;&nbsp;cin &gt;&gt; x;<br />
&nbsp;&nbsp;&nbsp;&nbsp;<br />
&nbsp;&nbsp;&nbsp;&nbsp;// ask user for a second number<br />
&nbsp;&nbsp;&nbsp;&nbsp;cout &lt;&lt; &quot;Enter a second number:&quot;;<br />
&nbsp;&nbsp;&nbsp;&nbsp;<br />
&nbsp;&nbsp;&nbsp;&nbsp;// User inputs Y value :: test for second variable and order<br />
&nbsp;&nbsp;&nbsp;&nbsp;cin &gt;&gt; y; </p>
<p>&nbsp;&nbsp;&nbsp;&nbsp;// adds totals and displays X+Y :: test for expression<br />
&nbsp;&nbsp;&nbsp;&nbsp;cout &lt;&lt; &quot;You entered a combined total of &quot; &lt;&lt; x+y &lt;&lt; std::endl;<br />
&nbsp;&nbsp;&nbsp;&nbsp;return 0;<br />
}</p>
<p>I like to test things like this by running several previous things I&apos;ve learned together to make sure they actually work the way I think they do. In this case, I discovered that the space at the end of &quot;Enter a number: &quot; in the original example is unnecessary in some cases; you can&apos;t see it anyway, since it&apos;s a command prompt and the only thing different would be a space between the : and your inputted integer. The space makes it a little prettier, but doesn&apos;t stand out as especially needed.</p>
<p>The second thing was actually the first chronologically discovered; the realization that when I put in:</p>
<p>&nbsp;&nbsp;&nbsp;&nbsp;std::cout &lt;&lt; &quot;You entered a combined total of&quot; &lt;&lt; x+y &lt;&lt; std::endl;</p>
<p>&#8230;it printed off &quot;You entered a combined total ofX&quot;</p>
<p>This meant I had to go searching for where the space was coming from, which turned out to be included within the end of the quotations. This also made me curious, since it means spaces are preserved in quotes, so does it minimize it to one space or does it count all spaces?</p>
<p>A simple test of adding 3 spaces to the end of the quotations so it&apos;d be obvious verifies that, yes, C++ does in fact preserve all spaces entered within quotations.</p>
<p>Combined with checking the printed data, and noting it has a monospaced font, this means it can be used to print off ASCII art. =P</p>
<p>Anyway, mostly irrelevant, but it did prove that the code works (mostly) the way I thought it did, and I was able to learn some neat things by testing some variations thereof.</p>
<p>For any newbies reading, I&apos;d generally recommend running the presented text first (write it out manually instead of copy/paste, helps you to remember the commands via repetition), and then do something a little different to test how it works in practice under new conditions. It helps to make sure you understand what&apos;s happening and why it&apos;s working, and if something breaks during the test, it means you did something wrong in an enclosed environment where you can&apos;t break too much.</p>
<p>Better to screw up while learning in a relatively safe area than screw up later on when getting paid and your boss gets pissy when you break it and spend an hour debugging something you could&apos;ve learned early on. Learning involves making mistakes; use your learning time to get all the mistakes out of the way by feeling out what works and what doesn&apos;t before it matters. =3</p>
<p>Also, while I overdid it (many times over) with the comments, in past experience I&apos;ve learned spending 15 minutes writing out clear explanations for why you did something will save you hours of work later on when it invariably breaks and you can&apos;t recall what you were thinking at the time you wrote it. It&apos;s even more useful if someone else has to read your code or work because they may not come to the same conclusions you did, such as &quot;using std&quot; instead of &quot;using namespace&quot;. Any time there&apos;s a judgement call, give a clear explanation, it&apos;ll save headaches in the long run, even if it feels like a waste of time now.</p>

		<p>Both cin and cout live inside the std namespace.  If you use &#8220;cin&#8221; without any prefix, the C++ compiler won&#8217;t assume you mean std::cin.  You either have to use std::cin, or use a &#8220;using&#8221; statement so the compiler knows that cin really means std::cin.</p>
<p>So I learned something new about Visual Studio today.<br />
* If you run your program normally (&#8220;Start Without Debugging, keyboard shortcut ctrl-F5), Visual Studio will automatically pause at the end of your application.<br />
* If you run your program with debugging (&#8220;Start Debugging&#8221;, keyboard shortcut F5), Visual Studio will not pause at the end of your application.</p>
<p>My suggestion is simply to run your programs without debugging, unless you need to debug.  If you do need to debug, you can put a breakpoint on the closing brace of main() to ensure it stops.</p>
